# Scope Pi — OpenClaw Setup

Everything in this folder is personal configuration for running OpenClaw on a
Raspberry Pi from this fork. Nothing here is part of upstream `openclaw/openclaw`.

## What's in here

```
pi/
├── README.md          # this file
├── bootstrap.sh       # idempotent: writes ~/.openclaw/openclaw.json from scratch
├── env.example        # template for ~/.openclaw/.env (secrets)
└── workspace/         # agent workspace (AGENTS.md, SOUL.md, BOOT.md, …)
```

The Pi runs three separate pieces of state, and only the repo is git-managed:

| Thing | Path on the Pi | Git-managed? |
|---|---|---|
| OpenClaw source | `~/Developer/openclaw` (this clone) | yes (fork) |
| Runtime config | `~/.openclaw/openclaw.json` | no (generated by `bootstrap.sh`) |
| Secrets | `~/.openclaw/.env` | no (per-machine only) |
| Agent workspace | `pi/workspace/` inside this repo | yes (this fork) |

Because the workspace lives inside the repo, editing `AGENTS.md`, `SOUL.md`,
`BOOT.md`, `PR_FROM_SLACK.md`, etc. from your laptop and running `git pull` on
the Pi is the whole update loop. No re-onboarding needed.

Target GitHub repos that the bot opens PRs against are cloned under
`pi/workspace/projects/<repo>/`. That directory is gitignored inside this fork
so the clones never leak back into your own repo history.

## First-time setup on the Pi

Run these once on the Pi after a fresh OS install.

### 1. Install OpenClaw

```bash
curl -fsSL https://openclaw.ai/install.sh | bash
openclaw --version
```

### 2. Clone this fork

```bash
mkdir -p ~/Developer
cd ~/Developer
git clone https://github.com/inspect-scope/openclaw.git
cd openclaw
```

### 3. Add the official repo as an upstream remote

This lets you pull new OpenClaw releases without pushing to the official repo.

```bash
git remote add upstream https://github.com/openclaw/openclaw.git
git remote set-url --push upstream DISABLE
git fetch upstream
```

Verify:

```bash
git remote -v
# origin     https://github.com/inspect-scope/openclaw.git (fetch)
# origin     https://github.com/inspect-scope/openclaw.git (push)
# upstream   https://github.com/openclaw/openclaw.git      (fetch)
# upstream   DISABLE                                       (push)
```

### 4. Create the secrets file

```bash
mkdir -p ~/.openclaw
umask 077
cp ~/Developer/openclaw/pi/env.example ~/.openclaw/.env
chmod 600 ~/.openclaw/.env
nano ~/.openclaw/.env          # fill in real values
```

At minimum: `GOOGLE_API_KEY`, `SLACK_BOT_TOKEN`, `SLACK_APP_TOKEN`,
`NOTION_API_KEY`, `OPENAI_API_KEY`.

### 5. Run OpenClaw's onboarding (creates the base config)

```bash
openclaw onboard
```

Pick any defaults. The wizard writes a minimal `~/.openclaw/openclaw.json`. We
overwrite the important fields in the next step, so don't worry about getting
the wizard's answers perfect.

### 6. Apply the Pi bootstrap

```bash
~/Developer/openclaw/pi/bootstrap.sh
```

This runs a series of `openclaw config set` commands that:

- Point the agent workspace at `~/Developer/openclaw/pi/workspace`
- Enable Slack (Socket Mode), Notion, Whisper, Google web search
- Replace every plaintext credential with a SecretRef to an env var
- Enable `session-memory` and `boot-md` hooks

The script is idempotent — re-run it any time you change the list of
integrations.

### 7. Install `gh` and authenticate

This is what lets the agent open GitHub PRs.

```bash
sudo apt update
sudo apt install -y gh git

# Option A — put a token in ~/.openclaw/.env (non-interactive):
#   GITHUB_TOKEN=<PAT with `repo` scope (or fine-grained equivalent)>
# Then just verify:
gh auth status

# Option B — one-time interactive login:
gh auth login        # pick HTTPS, authenticate with browser or a PAT

# Minimum git identity for commits the agent makes on the Pi
git config --global user.name  "Scope Pi Agent"
git config --global user.email "<your-github-noreply-address>"
```

Pre-clone the repos you expect the agent to open PRs against, so it doesn't
fight permissions on first use:

```bash
mkdir -p ~/Developer/openclaw/pi/workspace/projects
cd ~/Developer/openclaw/pi/workspace/projects
gh repo clone <owner>/<repo>
```

### 8. Verify everything

```bash
openclaw secrets audit --check      # expect 'clean'
openclaw gateway restart
openclaw gateway status --deep      # expect running
openclaw channels status --probe    # expect Slack 'works'
openclaw skills list | grep -E 'notion|github'  # expect both listed
gh auth status                      # expect logged in
```

Send a Slack DM to the bot to confirm round-trip.

## Daily edit loop

### Editing agent behavior (md files in `pi/workspace/`)

Edit on your laptop → commit → push → `git pull` on the Pi.

```bash
# laptop
nvim pi/workspace/AGENTS.md
git add pi/workspace/ && git commit -m "pi: tighten slack reply rules" && git push

# pi
cd ~/Developer/openclaw && git pull
```

No restart required. `AGENTS.md`, `SOUL.md`, `USER.md`, `IDENTITY.md`, `TOOLS.md`
are injected at the start of every new session. `BOOT.md` is read on every
gateway startup.

### Adding a new integration or flipping a config key

1. Edit `pi/bootstrap.sh` to add the new `openclaw config set …` line.
2. Commit and push.
3. On the Pi: `git pull && ./pi/bootstrap.sh && openclaw gateway restart`.

### Rotating a secret

```bash
nano ~/.openclaw/.env          # update the value
openclaw secrets reload        # re-resolve without restart
```

### Adding or removing an env var

1. Update `pi/env.example` so the template matches reality.
2. If a new env var is referenced in `bootstrap.sh`, the gateway will fail
   fast at startup if it's missing — that's the fail-fast SecretRef behavior.
   Add the var to `~/.openclaw/.env` first, then run bootstrap.

## Syncing with official upstream

`main` stays a pure mirror of `upstream/main`. All personal work lives on
`personal/pi`.

### Initial branch layout (one-time)

```bash
git checkout main
git fetch upstream
git reset --hard upstream/main
git push origin main                  # mirror to your fork

git checkout -b personal/pi           # long-lived branch for Pi work
git push -u origin personal/pi
```

Move the `pi/` folder onto `personal/pi` (it already lives there in this fork).

### Weekly sync

```bash
# 1. Update the mirror
git checkout main
git fetch upstream
git merge --ff-only upstream/main     # must fast-forward
git push origin main

# 2. Rebase your Pi branch on top of the new upstream
git checkout personal/pi
git rebase main
git push --force-with-lease origin personal/pi
```

If the rebase hits conflicts outside `pi/`, stop — that means you're editing
upstream files on `personal/pi`, which defeats the point. Move those changes
to a short-lived `feat/...` branch off `main` and open a PR to upstream
instead.

### On the Pi after a sync

```bash
cd ~/Developer/openclaw
git checkout personal/pi
git pull --ff-only
./pi/bootstrap.sh                     # only if bootstrap.sh changed
openclaw gateway restart              # only if core code changed
```

## Verifying Slack and Notion

### Slack

```bash
openclaw channels status --probe
openclaw channels capabilities --channel slack
openclaw channels logs --channel slack --lines 200
```

Healthy output shows `works` or `audit ok` for the Slack account. `probe failed`
means the bot or app token is wrong; the log lines will say which.

Round-trip test: DM the bot in Slack and watch
`~/.openclaw/agents/<agentId>/sessions/*.jsonl` for the new session.

### Notion

```bash
openclaw secrets audit --check                   # confirms NOTION_API_KEY resolves
openclaw skills list | grep notion               # confirms skill is loaded
openclaw message send "Use the notion skill to search for a page titled Hello"
```

If `audit --check` reports `REF_UNRESOLVED` for a Notion target, the env var
isn't visible to the gateway process. On systemd, add
`EnvironmentFile=%h/.openclaw/.env` to the unit.

### Slack → GitHub PR round-trip

This is the happy-path test. It proves Slack input, agent reasoning, and
GitHub output all work end-to-end.

On the Pi, once (pick any repo you own; a throwaway is fine):

```bash
cd ~/Developer/openclaw/pi/workspace/projects
gh repo clone <your-user>/<test-repo>
```

Then from Slack, DM the bot something like:

> Open a PR on `<your-user>/<test-repo>` that adds a file called `HELLO.md`
> with the text "hello from the Pi". The PR intent is "smoke test the Slack
> to GitHub pipeline".

Expected behavior (see `pi/workspace/PR_FROM_SLACK.md` for the full runbook):

1. Bot acknowledges in Slack.
2. Bot creates branch `agent/<date>-<slug>` in `projects/<test-repo>/`.
3. Bot commits the file, pushes, and runs `gh pr create` with a structured
   body (Intent / What changed / How to verify / Out of scope).
4. Bot replies in Slack with one line: `Opened <owner>/<repo>#<num> → <url>`.

Check the result:

```bash
gh pr list --repo <your-user>/<test-repo> --state open
gh pr view <num> --repo <your-user>/<test-repo>
```

If it fails, the most likely causes in order:

- `gh auth status` not OK on the Pi → refresh token or re-run `gh auth login`.
- No git identity set → `git config --global user.name`/`.email` (see step 7).
- Repo not cloned under `pi/workspace/projects/` → the agent won't invent a
  path; tell it the clone path explicitly or clone it yourself first.
- Slack reply never arrives → `openclaw channels logs --channel slack --lines 200`
  and `~/.openclaw/agents/<agentId>/sessions/*.jsonl` will show where it stopped.

## Troubleshooting

| Symptom | Likely cause | Fix |
|---|---|---|
| `REF_UNRESOLVED` in audit | `.env` not readable by gateway user | `chmod 600 ~/.openclaw/.env`, ensure ownership |
| Slack `probe failed` | bad token, or Socket Mode off on the app | regenerate app-level token with `connections:write`, update `.env`, `openclaw secrets reload` |
| Agent doesn't see workspace edits | workspace path wrong | `openclaw config get agents.defaults.workspace` |
| `BOOT.md` not running | `hooks.internal.enabled=false` | rerun `./pi/bootstrap.sh` |
| Upstream rebase full of conflicts | personal changes bleeding outside `pi/` | move them to a `feat/...` branch off `main`, PR upstream |
| Gateway won't start after `git pull` | dependencies or build out of date | `pnpm install && pnpm build && openclaw gateway restart` |
| Bot refuses to open PR / says it can't use `gh` | `gh` not installed or not on the gateway process `PATH` | `sudo apt install gh`, then ensure the gateway unit inherits `PATH` |
| `gh pr create` fails with `403` | token scope too narrow | regenerate PAT with at least `repo` (and `workflow` if touching Actions), update `GITHUB_TOKEN` in `.env`, `openclaw secrets reload` |
| Bot commits as `root` or blank author | git identity missing on the Pi | `git config --global user.name` / `.email` |

Useful one-liners:

```bash
openclaw config file                  # prints active config path
openclaw doctor                       # generic health check + fix hints
openclaw gateway logs                 # tail gateway logs
```
